## 1. RAG 概述与原理

### 1.1 什么是 RAG？

**RAG (Retrieval-Augmented Generation)**：检索增强生成，将外部知识检索与生成式模型结合的系统。

### 1.2 为什么需要 RAG？

| 问题 | LLM 局限 | RAG 解决方案 |
|------|----------|-------------|
| 知识过时 | 训练数据有截止日期 | 实时检索外部知识 |
| 幻觉 | 可能生成错误内容 | 基于真实文档生成 |
| 长尾知识 | 不常见知识覆盖少 | 精确检索补充 |
| 可溯源性 | 不知道答案来源 | 提供引用来源 |

---

## 2. RAG 架构与流程

### 2.1 完整 RAG 流程

```
用户查询 → 检索阶段 → 生成阶段 → 答案(附引用)
```

### 2.2 各阶段详解

| 阶段 | 输入 | 输出 | 关键技术 |
|------|------|------|----------|
| 文档加载 | PDF/HTML/Markdown | 原始文本 | PyMuPDF, Unstructured |
| 文本切分 | 长文本 | Chunks | 固定大小/语义切分 |
| 向量化 | Chunk | Vector | Embedding Model |
| 向量存储 | Vector | Index | FAISS, Milvus, Pinecone |
| Query改写 | 用户Query | 检索Query | HyDE, Query Expansion |
| 检索 | Query Embedding | Top-K Chunks | ANN, BM25 |
| 重排 | Top-K Chunks | 重排后列表 | Cross-Encoder |
| 生成 | Query + Context | 答案 | LLM |

## 3. 分块策略 (Chunking)

### 3.1 为什么分块重要？

分块直接影响检索质量和生成效果：
- **块太大**：引入噪声，上下文溢出
- **块太小**：丢失语义完整性，检索召回低

### 3.2 分块策略对比

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **固定大小** | 按 token/字符数切分 | 简单高效 | 语义可能断裂 |
| **语义切分** | 按句子/段落切分 | 保持语义 | 实现复杂 |
| **递归切分** | 多级粒度切分 | 灵活 | 调参复杂 |
| **文档结构** | 按标题/章节切分 | 符合写作逻辑 | 依赖文档格式 |

### 3.3 分块大小选择

| 应用场景 | 建议块大小 | 理由 |
|----------|-----------|------|
| 摘要 | 2,000-4,000 | 需要足够上下文 |
| 问答 | 500-1,000 | 精确匹配 |
| 搜索 | 256-512 | 快速检索 |
| 代码 | 128-256 | 函数级完整 |

## 4. 检索技术

### 4.1 稀疏检索 vs 密集检索

| 类型 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| **Sparse (BM25)** | 关键词匹配 + TF-IDF | 可解释、精确匹配 | 无法语义理解 |
| **Dense (Vector)** | 神经网络嵌入 | 语义相似 | 需要足够数据 |

### 4.2 BM25 算法

**Okapi BM25** 公式：

$$
\text{score}(D, Q) = \sum_{i=1}^{n} \text{IDF}(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{\text{avgdl}})}
$$

其中：
- $f(q_i, D)$：term $q_i$ 在文档 $D$ 中的频率
- $|D|$：文档长度， $\text{avgdl}$：平均文档长度
- $k_1$：词频饱和参数（通常 1.2-2.0）
- $b$：文档长度归一化参数（通常 0.75）
- $\text{IDF}(q_i) = \log \frac{N - n(q_i) + 0.5}{n(q_i) + 0.5}$

### 4.3 密集检索 (Dense Retrieval)

**双编码器架构**：

$$
\text{score}(q, d) = \text{cosine}(E_q(q), E_d(d))
$$

常用模型：BERT-base、DPR、BGE、M3E

### 4.4 HNSW 向量索引

- 构建多层图，每层是不同精度的近似
- 搜索时从顶层开始，逐层精化
- 时间复杂度： $O(\log N)$

## 5. 混合检索与 Rerank

### 5.1 混合检索 (Hybrid Search)

结合稀疏检索和密集检索的优势：

$$
\text{score}_{\text{hybrid}} = \alpha \cdot \text{score}_{\text{BM25}} + (1-\alpha) \cdot \text{score}_{\text{vector}}
$$

### 5.2 RRF (Reciprocal Rank Fusion)

**公式**：

$$
\text{RRF}(d) = \sum_{i=1}^{K} \frac{1}{k + \text{rank}_i(d)}
$$

其中 $k$ 是平滑因子（通常 $k=60$）。

### 5.3 Rerank 流程

```
初始检索 (Top-100) → Cross-Encoder 重排 → 最终 Top-10
```

**Cross-Encoder 优势**：同时编码 Query 和 Document，交互建模更精确。

## 6. RAG 优化策略

### 6.1 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 检索不到相关内容 | 块太大/嵌入不准确 | 调整 chunk size，使用更优嵌入 |
| 上下文过长 | Top-K 过多 | 上下文压缩、Self-RAG |
| 格式错误 | Prompt 不清晰 | 约束输出格式 |
| 回答不全面 | 召回不足 | 增加检索多样性 |
| 幻觉 | LLM 忽视检索内容 | 强制引用、Chain-of-Verification |

### 6.2 Query 改写技术

**HyDE (Hypothetical Document Embeddings)**：
1. LLM 生成假设性回答
2. 用回答的 embedding 检索
3. 返回真实相关文档

### 6.3 Self-RAG

Self-RAG 通过**反思 token** 自评估检索和生成质量：

| Token | 作用 |
|-------|------|
| `[检索]` | 是否需要检索 |
| `[相关]` | 检索内容是否相关 |
| `[完整]` | 回答是否完整 |
| `[支持]` | 回答是否被检索内容支持 |

## 7. RAG vs SFT

### 7.1 核心区别

| 维度 | RAG | SFT |
|------|-----|-----|
| 知识更新 | 动态更新知识库 | 需要重新训练 |
| 推理成本 | 检索 + 生成 | 直接推理 |
| 适用场景 | 需要最新知识 | 固定模式/风格 |
| 幻觉风险 | 可控（基于检索） | 依赖模型 |

### 7.2 何时用 RAG vs SFT？

**选择 RAG**：
- 知识需要频繁更新
- 需要可溯源的答案
- 长尾知识覆盖
- 垂直领域知识库

**选择 SFT**：
- 固定的输出格式/风格
- 领域推理模式复杂
- 需要模型"学会"某种能力

## 8. 高频面试题

### Q1: RAG 的工作原理是什么？

**参考答案**：

RAG (Retrieval-Augmented Generation) 由检索和生成两部分组成：

1. **检索阶段**：用户 Query 向量化，在向量数据库中检索 Top-K 相关文档
2. **生成阶段**：将检索到的文档与 Query 组合成 Prompt，LLM 基于上下文生成答案

---

### Q2: RAG 的核心组件有哪些？

**参考答案**：

| 组件 | 功能 | 关键技术 |
|------|------|----------|
| 文档加载 | 读取各种格式 | PyMuPDF, Unstructured |
| 文本分块 | 切分文档 | 固定大小/语义切分 |
| 向量化 | 文本→向量 | BGE, M3E |
| 向量存储 | 索引管理 | FAISS, Milvus |
| 检索器 | 找到相关文档 | BM25, ANN |
| 生成器 | LLM 生成答案 | GPT-4, LLaMA |

---

### Q3: 分块策略有哪些？ 如何选择？

**参考答案**：

| 策略 | 适用场景 |
|------|----------|
| 固定大小 | 通用，简单高效 |
| 语义切分 | 保持语义完整 |
| 递归切分 | 多级粒度 |
| 文档结构 | 符合写作逻辑 |

**选择建议**：问答系统 256-512 tokens，摘要 2,000-4,000 tokens。

---

### Q4: 什么是混合检索？

**参考答案**：

混合检索结合**稀疏检索（BM25）**和**密集检索（向量相似度）**：

$$
\text{score} = \alpha \cdot \text{BM25}(Q, D) + (1-\alpha) \cdot \text{cosine}(E_Q(Q), E_D(D))
$$

稀疏检索擅长精确关键词匹配，密集检索擅长语义理解，二者互补。

---

### Q5: RAG 和 Agent 的关系？

**参考答案**：

**RAG 是 Agent 的工具之一**。Agent 可以使用 RAG 获取知识，但 Agent 还具备规划、工具调用、反思等更全面的能力。

---

### Q6: 如何解决 RAG 的检索召回率低问题？

**参考答案**：

| 方法 | 描述 |
|------|------|
| Query 改写 | HyDE、Query Expansion |
| 混合检索 | BM25 + Vector |
| 增加索引 | 多种 Embedding |
| 元数据过滤 | 提前过滤不相关 |
| 知识图谱 | GraphRAG 增强 |

## 📝 面试高频题 & 参考答案

### Q1: RAG 和 SFT 的区别是什么？何时用 RAG 何选用 SFT？

**参考答案：**

| 维度 | RAG | SFT |
|------|-----|-----|
| **知识更新方式** | 外部检索，实时更新 | 需要重新训练/微调 |
| **知识正确性** | 可溯源，有据可查 | 依赖训练数据质量 |
| **实现复杂度** | 中（需维护检索系统） | 中（需训练流程） |
| **幻觉问题** | 较轻（基于检索结果生成） | 较重（模型自己编） |
| **推理延迟** | 高（多了检索步骤） | 低（纯生成） |
| **适用场景** | 知识频繁更新、需引用来源 | 任务格式/风格固定 |

**选择原则**：
- 知识需要**实时/频繁更新** → RAG
- 需要**强格式遵循**（如 JSON 输出）→ SFT
- **两者结合**效果通常更好：RAG 提供知识，SFT 塑造格式

---

### Q2: RAG 的核心流程是什么？

**参考答案：**

```
用户问题 → Query 改写 → 向量检索（Top-K）→ 文档重排 → 上下文组装 → LLM 生成 → 最终答案
     ↑                                                                              ↓
     ←——————————————————————————— 引用标注 & 置信度反馈 ————————————————————————————
```

**关键环节说明**：

1. **Query 改写**：将口语化/简略的查询扩展为更适合检索的形式（如隐式问题显式化）
2. **向量检索**：用 Embedding 模型将 query 和文档块编码为向量，余弦相似度召回 Top-K
3. **Rerank**：用 Cross-Encoder 对 Top-K 结果重新排序，选最相关的几条
4. **上下文组装**：将 rerank 后的文档块拼接为 prompt，控制 token 数量

---

### Q3: RAG 常见的失败场景有哪些？如何解决？

**参考答案：**

| 失败场景 | 原因 | 解决思路 |
|---------|------|---------|
| **内容缺失** | 切片策略不合理、覆盖不全 | 调整 chunk size；多向量检索；增加索引质量评估 |
| **错过排名靠前** | 向量相似度计算方式不优 | cosine → dot-product；引入 rerank |
| **脱离上下文** | 多文档拼接时语义边界丢失 | 语义分块；Context Window Re-weighting |
| **未能提取答案** | 检索结果太宽/太窄 | 调整相似度阈值；多跳检索 |
| **格式错误** | 数据清洗不规范；Prompt 约束不足 | 清洗 HTML/Markdown；强化 Prompt 格式约束 |
| **答案不完整** | 需要多文档综合，单文档不够 | HyDE（生成假设答案再检索）；多跳推理 |

### Q4: 如何评估 RAG 系统的效果？

**参考答案：**

| 评估维度 | 指标 | 说明 |
|---------|------|------|
| **检索质量** | Recall@K, MRR | 检索到的文档是否真正相关 |
| **生成质量** | RAGAS, ROUGE, BLEU | 生成答案与参考答案的 overlap |
| **Faithfulness** | 幻觉率 | 答案是否被上下文支持 |
| **引用准确性** | Citation Precision | 引用的文档是否真的支撑答案 |
| **端到端** | Task Success Rate | 用户问题是否被正确回答 |

**RAGAS（无参考评估）**：
- Faithfulness：根据上下文评估答案是否忠实于检索内容
- Answer Relevancy：答案与问题的相关程度

### Q5: 什么是 Hybrid Search？为什么需要它？

**参考答案：**

Hybrid Search = **向量检索（Semantic） + 关键词检索（BM25）**

**为什么需要**：
- 向量检索擅长捕捉语义相似性，但**专有名词/精确术语**检索能力弱
- BM25 是基于词频的精确匹配，擅长**关键词召回**

**实现方式**：
```python
# 两种召回结果合并
vector_results = vector_db.search(query, top_k=20)
bm25_results = bm25_index.search(query, top_k=20)

# 方法1：RRF（Reciprocal Rank Fusion）
# 方法2：加权得分（alpha * vector_score + (1-alpha) * bm25_score）
merged = rrf_fusion(vector_results, bm25_results, k=60)
```

### Q6: 文档切片（Chunking）有哪些策略？各有何优缺点？

**参考答案：**

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **固定长度分块** | 按 token 或字符数硬切 | 简单可控 | 可能切在句子中间，丢失语义 |
| **句子级别分块** | 按句号/换行切分 | 语义完整 | chunk 大小不均 |
| **段落级别分块** | 按段落切分 | 语义连贯 | chunk 可能过长 |
| **语义分块** | 用模型判断句子边界 | 语义最完整 | 成本高、速度慢 |
| **Parent Document** | 小块精确匹配 + 父文档补充完整上下文 | 兼顾精确和完整 | 实现复杂 |

**chunk size 选择**：通常 256~512 tokens；简单问答偏小（128~256）；复杂分析偏大（512~1024）

### Q7: 什么是 Query Expansion / Rewrite？为什么重要？

**参考答案：**

**Query Rewrite**：将用户的原始问题改写为更适合检索的形式。

**常见方法**：
1. **隐式问题显式化**：`"它的副作用"` → `"某药物的副作用"`
2. **口语转书面**：`"股票能买吗"` → `"某股票投资价值分析"`
3. **拆分为多查询**：`"Transformer原理和GPT区别"` → `["Transformer原理", "GPT架构"]`
4. **HyDE**：先生成假设答案，再以假设答案去检索（适合模糊查询）

**重要性**：用户的输入往往简短、模糊、有歧义，直接检索效果差；Query Rewrite 能显著提升召回率。

---

### Q8: 什么是 Embedding Model？RAG 中如何选择？

**参考答案：**

Embedding Model 将文本编码为稠密向量，使得语义相似的文本在向量空间中接近。

**RAG 中的选择考量**：
1. **维度**：768 维通常比 384 维效果好，但存储/检索成本高
2. **多语言支持**：中文场景需选中文优化模型（如 BGE、m3e）
3. **稠密 vs 稀疏**：BGE 是主流稠密向量；BM25 是经典稀疏向量（不需要模型）
4. **微调**：在领域数据上微调 Embedding 可显著提升领域召回率

**常见中文 Embedding 模型**：BGE-large-zh、m3e-large、text2vec-large、Jina-embeddings-v2
